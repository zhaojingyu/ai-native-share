# AI Native 组织协作分享 · 讲师备注（v2）

> 一句话定位：一次 AI Native 组织的探索。
>
> 时长预算 25–35 分钟，共 8 个主段落 + Q&A。每段标题后是「讲什么」，不是「念什么」。

---

## 0. 开场（1 分钟）

不要从「AI Native」这个词开始。直接抛问题：

> 「大家都在用 AI 了，但团队真的变快了吗？」
>
> 这场分享给一个可能的原因：个人提速了，但组织的协作方式没跟上。

---

## 1. 核心判断：单点提效 ≠ 组织提速（3 分钟）

**金句 1**：组织效率不是所有人的个人效率相加。

$$\text{组织效率} = \text{节点处理能力} \times \text{节点间流通速度}$$

**关键点**：瓶颈通常在交接处，不在工位上。
更要命的是——**每个人对项目的记忆没有版本管理**，思考过程和核心决策无法回溯，
人走了，知识也走了。

**金句 2**：组织提速靠的是「节点能力 × 流通速度」的同时提升。

---

## 2. 核心类比：从马车到汽车（2 分钟）

从马车到汽车，不是把马换成发动机就结束了。

| 马车时代 | 汽车时代 | AI Native 组织 |
|---|---|---|
| 人骑马 | 人开车 | 人 + agent |
| 土路 | 公路系统 | git · 版本化事实源 |
| 个人骑术 | 交通规则 | spec / plan / delivery · 文档协议 |
| 沿河取水 | 加油维修网络 | superpowers + skills + 运行手册 |
| 自负其责 | 驾照 + 保险 | 治理 + 二次确认 + 审计 |

**讲点**：真正支撑汽车时代的是一整套基础设施。重点不是 agent，是 agent 周围那套系统。

---

## 3. 信息通路三件套（4 分钟）

**这是整场分享的关键工程洞察**：

> **git · 钉钉 doc MCP · 钉钉群机器人** 三件套共同组成 AI Native 组织的高速信息通路。

每一件都不可替代：

- **git** → 版本化事实源，可 diff、可回溯、可审计
- **钉钉 doc MCP** → 私有配置、凭据、token、文档协议层（不适合进 git 的东西都放这）
- **钉钉群机器人** → 实时广播 + 流转下一棒（项目现场的实时神经系统）

**讲点**：少一个，整条链路就会回到口头协作。
- 没 git → 不可回溯
- 没 doc MCP → 私有信息无处放
- 没群机器人 → 完成事项无人知

---

## 4. 六个设计原则（8 分钟，每个 1–1.5 分钟）

1. **事实源前置**：需求、边界、验收、决策进 spec / plan / delivery / TODO。聊天只触发，不沉淀。
2. **每个人带一个 agent**：人负责判断和授权；agent 负责吞吐量。
3. **agent 通过文档高速交接**：借助 superpowers 等 skill，agent 不只产出 spec 和 plan，还能把"为什么这么决策"沉淀成可追溯的**思考和决策文档**。
4. **IM 用来加速，不用来沉淀**：钉钉群是项目现场；机器人广播交付和待办。
5. **三件套组成信息通路**：git + 钉钉 doc MCP + 群机器人，缺一即断。
6. **提升流速，而不是制造更多动作**：看 Spec→Delivery 周期、交接等待、返工率。

**金句 3**：人保留判断权和责任，agent 承担吞吐量。

---

## 5. 协作闭环（3 分钟）

五步闭环（注意每一步都留 commit + 测试 + delivery 痕迹）：

```
01 人提目标 + 自己的 agent 写 spec
02 spec 进 git，成为正式交接物
03 接手方 agent 读 spec、生成 plan、提澄清问题
04 agent 执行 / 测试 / 提交 / 群里广播
05 下一个人带 agent 接力，不重听背景
```

**讲点**：传统模式交接靠「人在线 + 记忆」；AI Native 模式交接靠「文档 + git」。

## 6. 实际案例 · 三个真实画面（5 分钟）

> 这一段是分享的视觉高潮，让方法论落到画面上。

### 画面 1：钉钉群机器人推送

`@jyzhao` 提交了 admin quota Phase A/B 需求，机器人自动 at：
- `@sy.hang` → 下一步：产品验收 portal 登录流程
- `@july.zhou` → 下一步：技术调研 new-api 方案
- `@sy.hang` → ⚠ 阻塞：portal 用户登录失败

**讲点**：提交者写明交付了什么，at 到具体的人 + 具体的下一步 TODO。
下一个协作人不用再问「我该做什么」。

### 画面 2：git 提交记录

时间线呈现：谁在哪个时间点做了什么。

```
06-12 14:30  @sy.hang  写 spec · admin quota 边界
06-12 16:45  @jyzhao    DB 变更 · corp / team_quota
06-12 18:20  @jyzhao    UI 调整 · admin 后台
06-12 19:02  @jyzhao    Shell · weekly_cap，提测
```

**讲点**：全部以原始版本的形式留在 git 里。
任何人、任何 agent 接手时都能拿到完整时间线和决策链。

### 画面 3：Agent 工作台

人只需要问一句「**帮我看下有什么需要我做的**」。
Agent 一次性列出：

- **上次提交**：别人做了什么（@jyzhao 提交 admin quota Phase A/B）
- **你的下一步**：产品验收 portal 登录流程
- **验收标准**：5 个测试账号全通过
- **下一棒是谁**：完成后流转给 @july.zhou

**讲点**：协作不再依赖记忆和口头确认，agent 直接把接力棒递到手里。

## 7. 评估方式（3 分钟）

不能用「AI 调用次数」「token 消耗量」「生成了多少代码」来判断。
这些指标只能说明「用了 AI」，不能说明「组织真的变强」。

### 北极星指标

> **在质量和风险不变差的前提下，单位时间内完成的有效业务交付量提升了多少。**

### 五维评估框架

| 维度 | 管理问题 | 可观察指标 |
|---|---|---|
| 业务产出 | 是否交付了更多有效结果 | 有效需求完成数、目标达成率、收入/降本影响 |
| 组织流速 | 信息和任务是否更快穿过组织 | Spec→Delivery 周期、交接等待时间、阻塞响应时间 |
| 质量稳定 | 快了之后有没有更乱 | 一次验收通过率、返工率、线上事故率、回滚次数 |
| 组织杠杆 | 同样人数是否产生更多高质量交付 | 人均有效交付、单位需求成本、会议时间减少 |
| 治理安全 | 自动化是否可追溯、可审计、可控 | 关键决策回写率、审批合规率、凭据泄露数 |

## 8. 落地路径（3 分钟）

四步把一个新项目初始化成 AI Native 模式：

### 阶段 A · 建群 + git 项目 + 初始化

- 新建项目钉钉群和 git 仓库
- 下载 [SOP 模板文档](./sop-template.md)，放进项目根目录
- 模板会自动初始化 spec / plan / TODO / delivery 目录结构
- **后续新项目重复这一步即可**

### 阶段 B · 每人配 Agent + skills

- 每个人把自己的 agent 接入项目
- 装 [superpowers](https://github.com/obra/superpowers) skill 集合
- 接 [钉钉 MCP](https://aihub.dingtalk.com/#/mcp) 让 agent 能读写文档、调用钉钉能力

### 阶段 C · 连机器人 + 建映射

- 在群里连接好钉钉机器人
- 做好三件事的映射：**钉钉机器人 ↔ 项目 git 地址 ↔ 群里相关人**
- 让 push、delivery、阻塞自动广播给正确的人

### 阶段 D · 发布任务 + 看指标

- 按 spec → plan → TODO → delivery 的闭环发布任务
- 用北极星指标和五维评估表持续优化
- 跑出真实数据再扩张到更多项目

## 9. 超越技术团队（2 分钟）

**讲点**：这场分享虽然写满了 git、commit、agent 这些词，但底层结构对任何岗位都成立。

### 三个层次

1. **术语是技术化的，结构是普适的**
   - 把 git 换成文档版本，把 commit 换成交付物，业务团队立刻能用。
2. **从 vibe coding 走向 vibe working**
   - agent 已经从「帮工程师写代码」走向「帮任何人完成结构化工作」。
   - 业务、运营、市场、客服都在长出自己的 agent 协作模式。
3. **未来：商业化的 cowork 产品**
   - 今天手工搭的 git + 钉钉 doc MCP + 群机器人链路，
     未来会被产品化、被普及、被更多非技术人员直接用上。

---

## 10. 结尾 + 下一步号召（1 分钟）

### 总结金句

> token 消耗有边际递减，真正的杠杆在 agent 之间的接力。
>
> 普及一定的人均 token 消耗量是基础，
> 但更重要的是把 agent 之间的接力做顺。

### 给观众的下一步

- **明天**：下载 SOP 模板，给一个新项目初始化。
- **下周**：装 superpowers + 钉钉 MCP，连群机器人。
- **一个月**：选一个项目跑试点，用北极星指标对比前后差异。

## Q&A 预案

| 可能的问题 | 我的回答要点 |
|---|---|
| agent 输出错怎么办？ | 验收标准前置 + 高风险动作二次确认 + delivery 摘要可审计。错是必然的，关键是错得快、错得可见、错得可回滚。 |
| 团队小值得这么搞吗？ | 越小越值得。三件套的边际成本对 3 人团队和 30 人团队是一样的，但流速收益对小团队放大更明显。 |
| 这套对探索性需求适用吗？ | 探索期文档可以更轻（一份问题清单 + 一份假设列表），但闭环结构不变。结构化 ≠ 重量化。 |
| 非技术团队能用吗？ | 能。把 git 换成钉钉文档版本，把 commit 换成交付物，三件套和闭环完全适用。 |
| 我们已经有 Jira / 飞书 / 企微，怎么办？ | 工具不重要，机制重要。只要满足「版本化事实源 + 文档协议层 + 实时广播」三件套，用什么工具都行。 |

## 演讲节奏提醒

- **节奏感**：每个原则讲 60 秒概念 + 30 秒案例，超过 1.5 分钟就停。
- **视觉锚点**：每抛一个金句，停 2 秒，让观众看清屏幕。
- **画面优先**：第 6 段案例是视觉高潮，让画面说话，不要照念。
- **行动感**：结尾不要只总结，给观众明天就能做的三件事。
- **诚意感**：第 9 段「超越技术团队」必须讲，这是这场分享视野的延伸，避免让人觉得这是「研发内部的事」。
